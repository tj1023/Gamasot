# 🥘 Gamasot (가마솥)

> **2D 캐주얼 덱빌딩 게임**<br>
> 재료를 수집하고 다양한 장식품의 시너지 효과를 결합해 최고 점수에 도전하는 게임입니다.<br>
> 아키텍처의 SOLID 원칙과 메모리 최적화를 중점적으로 고려하여 설계되었습니다.

## 🛠 기술 스택
- **Engine:** Unity 6.2
- **Language:** C#
- **Version Control:** Git

---

## 🌟 핵심 설계 및 강점

본 프로젝트는 **성능 중심의 최적화**와 새로운 기능 추가 시 코드 수정 없이 인스펙터 조작만으로 확장이 가능한 **데이터 주도 설계(Data-Driven Design)** 를 목표로 개발되었습니다.

### 1. 상태 및 결합도 제어
* **GameContext 분리:** 전역 상태가 거대한 싱글톤에 종속되는 문제를 막기 위해 인게임 데이터와 런타임 상태를 `GameContext` 객체로 완전히 분리했습니다. 이를 통해 모듈 간 결합도를 낮추고 테스트 및 유지보수가 쉬운 환경을 구축했습니다.
* **상태 기계(FSM)와 제네릭 Event Bus:**
  * 선택 - 발견 - 플레이 - 정산 등 복잡한 페이즈 전환 시 발생하는 흐름 제어 버그를 해결하고자 FSM을 도입했습니다.
  * Delegate로 인한 클로저 할당과 박싱/언박싱 비용을 원천 제거하기 위해 `struct` 타입의 **제네릭 이벤트 버스(`EventBus<T>`)** 를 자체 구현하여, 견고한 이벤트 파이프라인을 구축했습니다.

### 2. 데이터 주도 설계 및 확장성
* **Strategy 패턴과 SubclassSelectorDrawer:** 하드코딩된 거대한 Switch-case 로직으로 묶여있던 장식품 시스템을 `[SerializeReference]` 기반의 **Strategy 패턴**으로 분리했습니다.
  * 커스텀 에디터인 `SubclassSelectorDrawer`를 자체 제작하여, 개발자가 다형성 객체를 인스펙터에서 손쉽게 선택하고 추가할 수 있도록 개발 UX를 개선했습니다.
* **Command 패턴과 Synergy 시스템:** 플레이어의 액션과 효과 적용 로직을 Command 패턴으로 캡슐화했습니다. 이를 기반으로 특정 재료가 모였을 때 발동하는 `SynergyData` 처리 로직을 구현해 유지보수가 용이한 구조를 설계했습니다.
### 3. 메모리/성능 최적화
* **O(1) Swap-and-Remove Object Pool:**
  * 매 라운드마다 잦은 재료(Ingredient) 스폰/파괴 시 발생하는 GC 스파이크와 프레임 드랍을 방지하기 위해 자체 ObjectPool을 구축했습니다.
  * 리스트를 갱신할 때 `List.RemoveAt`의 O(N) 요소 밀어내기 연산 대신, 마지막 요소와 자리를 바꾸고 제거하는 **O(1) 성능의 Swap-and-Remove 기법**을 적용해 런타임 최적화를 이뤄냈습니다.
* **WaitCache:** 코루틴 실행 시 `new WaitForSeconds`로 인해 매번 발생하는 메모리 할당(GC)을 차단하기 위해, 시간별 지연 객체를 딕셔너리로 캐싱해 재활용하는 `WaitCache` 시스템을 도입했습니다.

## 💡 문제 해결

| 문제 상황 | 해결 과정 | 결과 |
| :--- | :--- | :--- |
| **강한 결합도와 하드코딩**<br>장식품 효과 추가 시 `TrinketManager`의 내부 코드 라인이 수백 줄씩 늘어남 | 구체적 시점(OnHarvest 등)의 로직을 `TrinketEffect` 상속 클래스(전략 패턴)로 위임하고 에디터 할당 기능 추가 | 기존 코드를 건드리지 않고(OCP 준수) 인스펙터 조작만으로 신규 효과 추가 가능 |
| **GC 스파이크와 프레임 저하**<br>재료 오브젝트와 코루틴 WaitForSeconds에서 잦은 힙 할당 발생 | `UnityEngine.Pool`을 활용한 자체 풀 구축 (Swap-and-remove 기법 적용) 및 `WaitCache` 캐싱 시스템 도입 | 핵심 게임 루프 중 런타임 GC 할당 최적화 |
| **복잡한 비동기 로직 충돌**<br>이벤트 다중 발생 시 다음 게임 페이즈로 넘어가지 못하고 멈추는 데드락 현상 | 콜백 체인을 지양하고 상태(FSM)와 통신(제네릭 Event Bus)을 완전히 분리해 순차적 Command Queue로 시너지 효과 처리 | 안정적인 파이프라인 형성 및 버그 해결 |
